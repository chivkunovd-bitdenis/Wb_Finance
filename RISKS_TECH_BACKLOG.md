# Реестр рисков и Tech Backlog

Документ предназначен для накопления “сложных” кейсов: риски согласованности, гонки, регрессии, и технические улучшения, которые не обязательно делать сразу.

Правило:
- при появлении нового риска/улучшения — добавляйте запись сюда, а не в `BACKLOG.md`.

---

## Формат записи

- `ID`: краткий идентификатор
- `Компонент`: UI/Backend/БД/Асинхронность
- `Серьезность`: Low/Medium/High
- `Сценарий`: что именно происходит по шагам
- `Эффект`: что увидит пользователь/что сломается
- `Текущая защита`: что уже сделано в коде
- `Почему риск вообще возможен`: корень проблемы
- `Рекомендации`: что улучшить
- `Статус`: Open / Mitigated / Closed

---

## Открытые записи

### R-OP-001: Гонка UI vs async `recalculate_pnl` (operational expenses)
- `ID`: R-OP-001
- `Компонент`: UI (React) + Backend (FastAPI) + Асинхронность (Celery)
- `Серьезность`: Medium
- `Сценарий`:
  1. Пользователь на вкладке `Опер. расходы` нажимает “Сохранить” (создание/редактирование записи) в `POST/PUT /dashboard/operational-expenses`.
  2. Backend сохраняет запись `operational_expenses` и **ставит в очередь** пересчёт витрин через `POST /sync/recalculate` (Celery), но возвращает ответ пользователю сразу (синхронного ожидания завершения пересчёта нет).
  3. До того как Celery успеет обновить витрину `pnl_daily.operation_expenses` и пересчитать `pnl_daily.margin`, UI выполняет `onRefresh()`, что триггерит `GET /dashboard/pnl`.
  4. Если `GET /dashboard/pnl` прочитает `pnl_daily`, которые ещё не обновились, UI отобразит **старые** значения `operation_expenses` и/или `margin` на основном `Дашборде`.
  5. Позже, когда Celery завершит пересчёт, данные станут корректными, но пользователь увидит это только на следующем refresh/перезагрузке (или при повторном заходе на страницу).
- `Эффект`:
  - В момент сразу после сохранения расходов пользователь может кратковременно увидеть несогласованные значения `operation_expenses/margin` на `Дашборде`.
  - Это не нарушает консистентность БД “навсегда”, но может выглядеть как “мигание” или “не сразу применилось”.
- `Почему риск вообще возможен`:
  - `POST/PUT /dashboard/operational-expenses` и пересчёт `pnl_daily` выполняются асинхронно относительно UI.
  - Celery воркер может отработать позже, чем UI успевает сделать следующий `GET /dashboard/pnl`.
- `Текущая защита`:
  - В `frontend/src/screens/OperationalExpenses.jsx` после `POST/PUT /dashboard/operational-expenses` UI:
    - вызывает `POST /sync/recalculate`;
    - затем делает polling через `GET /dashboard/pnl` по конкретной сохранённой дате, ожидая пока `operation_expenses` в `pnl_daily` станет равным ожидаемой сумме (допуск по округлению/погрешности);
    - только после этого вызывает `onRefresh()`.
  - Поллинг ограничен таймаутом ~60 секунд: если пересчёт не успевает, пользователь всё равно получит `onRefresh`, но может увидеть старые значения.
- `Рекомендации` (Tech Backlog):
  1. Добавить API-механизм ожидания завершения пересчёта:
     - например, возвращать Celery `task_id` из `/sync/recalculate`;
     - добавить `/sync/recalculate/status/{task_id}` и в UI ожидать факт обновления (без “слепого” polling по P&L).
  2. Рассмотреть “частичное” обновление `pnl_daily.operation_expenses` синхронно в момент изменения `operational_expenses`:
     - это уменьшит зависимость от полного пересчёта `recalculate_pnl` и радикально снизит риск гонки.
  3. Улучшить покрытие тестами:
     - добавить тест-юнит/интеграцию на кейс “дата содержит только `OperationalExpense`, без `RawSale/RawAd`” (частично уже покрыто на уровне unit/контракта).
  4. Добавить UI UX-обратную связь:
     - при ожидании пересчёта показывать “Применяем к марже…” (с прогрессом/таймером), чтобы не было ощущения “подвисло”.
- `Статус`: Mitigated (с polling), остаточный риск по таймауту сохраняется.
