## Уже реализовано (каркас)

- Импорт метрик отчёта (ручной + Playwright), список/детализация отчёта, `valid_until` / статусы, журнал `actions`.
- Парсер Excel (в т.ч. лист «Показатели», медиана по колонкам конкурентов в файле).
- Celery: скачивание нескольких периодов за один запуск (env), zip→xlsx.
- Ежедневная аналитика: порог 20% к медиане конкурентов, логистика из `sku_daily`, гипотезы/задачи с дедупом.
- UI ИИ-модуля: выбор товара, онбординг, WB-доступ (noVNC + файл), задачи/гипотезы, архив, журнал обновлений.
- Тестовый сидер данных по артикулу (дашборд/воронка) — **не подменяет** состав сравнения в кабинете WB.

## Ближайшие вертикали (рекомендуемый порядок)

1. **Конкурентные данные в БД (ТЗ 6–7)**  
   - Накопление импортов: батчи + аналитика по `latest_import_batch_id` (**сделано в коде** после миграции).  
   - Расширить парсер/схему: все нужные листы Excel, **медианная цена**, сырые колонки в JSON при необходимости.

2. **Акции (ТЗ 5 + 9.2.5)**  
   - Источник «активная акция» и цены по акции; расчёт маржи по правилам ТЗ; гипотеза «Вступить в акцию»; тексты «почему не вошли»; оповещения; авто-исключение (если есть API/действие в WB — отдельное исследование).

3. **Задачи на отчёт (ТЗ 7.4–7.5, 9.1)**  
   - Автозадачи «создать / обновить сравнение» при отсутствии/истечении `valid_until`, согласование расхода лимита с пользователем (уже частично через refresh + confirm).

4. **Полный паритет таблиц 3.2 / 4.2**  
   - Сверить каждый тип задачи/гипотезы с условиями ТЗ (в т.ч. социальные пороги для самовыкупов).

5. **ЖЦ гипотезы 10.5**  
   - Итог по периоду: сработала / нет / мало данных / нужен длиннее период (правила + UI).

## Важное про «тестовый товар в Excel»

Отчёт сравнения WB формируется **только из карточек, добавленных пользователем в сравнение в кабинете**. Данные сидера в нашей БД (`sku_daily` / `funnel_daily`) **не попадают** в файл Excel автоматически. Чтобы товар был в выгрузке, его `nm_id` должен быть в текущем сравнении на стороне WB.
