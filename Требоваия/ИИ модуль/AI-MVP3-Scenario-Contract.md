# AI-MVP3 — Scenario contract: ежедневная аналитика → задачи/гипотезы

## User action
- Пользователь инициирует “ежедневную аналитику” (в MVP3 — вручную через HTTP API).
- Пользователь затем открывает раздел “ИИ модуль → Задачи/Гипотезы” и видит новые записи.

## Expected visible result
- После запуска аналитики в системе появляются новые задачи/гипотезы по артикулам, где сработали правила.
- Повторный запуск аналитики на те же входные данные **не создаёт дубликаты** (idempotency / дедуп).

## System steps (backend)
- Авторизация пользователя.
- Загрузка конкурентного отчёта (AI-MVP2) по `report_id` в рамках `current_user`.
- Сбор метрик:
  - из отчёта конкурентов: `ctr`, `traffic`, `funnel_cart`, `funnel_order` (наши значения vs медиана конкурентов)
  - из `sku_daily`: логистика за `date_for` и среднее за последние 7 дней (если данные есть)
  - опционально из payload: `stock_days_left`, `social` (reviews/rating) для правил, где данных пока нет в БД
- Применение правил:
  - **Funnels**: если `funnel_cart` или `funnel_order` ниже медианы конкурентов на 20%+ → создать гипотезу `content_change`
  - **CTR**: если `ctr` ниже медианы конкурентов на 20%+ → создать гипотезу `ab_test`
  - **Self-buyouts**: если funnels ниже и `reviews < 13` или `rating < 4.4` (из payload) → создать задачу `self_buyouts`
  - **Logistics**: если логистика за день выше среднего 7 дней на 20%+ → создать задачи `check_measurements` и `check_ktr`
  - **Restock**: если `stock_days_left < 14` (из payload) → создать задачу `restock`
- Идемпотентность:
  - для задач/гипотез формируется стабильный `fingerprint`
  - при повторном запуске вместо дублей происходит “no-op” (или обновление поясняющих полей без смены статуса)

## Frontend call(s)
Пока UI не реализован, контракт фиксируем как HTTP API:
- POST `/ai/analytics/run`
- GET `/ai/tasks`
- GET `/ai/hypotheses`

## Backend endpoint(s)
- POST `/ai/analytics/run` — выполнить аналитику для пользователя
  - вход: `report_id`, опционально `date_for`, `stock_days_left`, `social`
  - выход: списки созданных `task_id` и `hypothesis_id` (только новые)

## Background process(es)
- В MVP3 — нет (запуск вручную). План: вынести в расписание/воркер позже.

## DB/state changes
- Создаются новые строки в `ai_tasks` и `ai_hypotheses` (status по умолчанию `new`/`draft`).
- Дедуп обеспечивается уникальностью `(user_id, fingerprint)`.

## What must NOT happen
- Пользователь не должен создавать/видеть результаты аналитики по чужому `report_id`.
- Повторный запуск аналитики не должен создавать дубликаты.
- Аналитика не должна сбрасывать/портить статус задач/гипотез, если пользователь уже начал работу.

## Happy-path verification
- Импортировать конкурентный отчёт (AI-MVP2) по одному артикулу с низкими funnels и CTR.
- Засидить `sku_daily` так, чтобы логистика “выросла”.
- Запустить POST `/ai/analytics/run` и убедиться:
  - созданы гипотезы `content_change` и `ab_test`
  - созданы задачи по логистике
  - при передаче `stock_days_left` и `social` создаются `restock` и `self_buyouts`
- Повторно запустить аналитику и убедиться, что дублей нет.

## Error/retry verification
- 401 без токена.
- 404 если `report_id` не существует или принадлежит другому пользователю.
- Повторный вызов endpoint безопасен (idempotent).
