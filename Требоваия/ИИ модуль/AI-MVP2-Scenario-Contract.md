# AI-MVP2 — Scenario contract: контур конкурентных данных (без Playwright)

## User action
- Пользователь вручную загружает (импортирует) конкурентный отчёт сравнения карточек (пока без Playwright).
- Пользователь просматривает список загруженных отчётов и открывает конкретный отчёт, чтобы увидеть метрики.

## Expected visible result
- После импорта отчёт появляется в списке “конкурентных отчётов”.
- При открытии отчёта пользователь видит набор метрик по артикулам (nm_id) с “наши значения” и “медиана конкурентов”.

## System steps (backend)
- Авторизация пользователя.
- Валидация payload (дата отчёта, период, допустимые metric_code).
- Идемпотентный импорт: один отчёт на `(user_id, report_date, period)`, повторный импорт обновляет метрики.
- Сохранение отчёта и метрик в БД.

## Frontend call(s)
Пока UI не реализован, фиксируем контракт как HTTP API:
- POST `/ai/competitor-reports/import`
- GET `/ai/competitor-reports`
- GET `/ai/competitor-reports/{report_id}`

## Backend endpoint(s)
См. выше; все требуют auth и работают только в рамках `current_user`.

## Background process(es)
- Нет. (Playwright job и актуальность 3 дня — это `AI-MVP4`.)

## DB/state changes
- Создаётся/обновляется `ai_competitor_comparison_reports` (уникальность по `(user_id, report_date, period)`).
- Создаются/перезаписываются строки `ai_competitor_metrics` для отчёта.

## What must NOT happen
- Пользователь не должен видеть чужие отчёты/метрики.
- Повторный импорт не должен создавать дубликаты метрик (должна быть замена/обновление).
- Нельзя принять неизвестный `metric_code`.

## Happy-path verification
- Импортировать отчёт с 2 метриками.
- Убедиться, что он появился в списке.
- Открыть отчёт и увидеть метрики.
- Повторно импортировать отчёт на ту же дату/период и убедиться, что метрики обновились без дублей.

## Error/retry verification
- 401 без токена.
- 400 при неверном `period` или `metric_code` или `nm_id`.
- 404 при запросе отчёта по `report_id`, который не существует или принадлежит другому пользователю.
