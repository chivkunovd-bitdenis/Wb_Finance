# Как обновить код на сервере

Когда ты что-то поменял в проекте и запушил в Git — чтобы то же самое работало на сервере, на сервере нужно подтянуть изменения и пересобрать контейнеры.

---

## Вариант 1: Деплой через Git (если проект уже клонирован на сервере)

### На своём компьютере

Закоммить и запушить изменения в репозиторий (GitHub / GitLab и т.п.):

```text
cd /Users/deniscivkunov/wb-finance
git add .
git commit -m "описание изменений"
git push
```

### На сервере

1. Зайти по SSH:
   ```text
   ssh root@194.87.96.144
   ```

2. Перейти в папку проекта и подтянуть код:
   ```text
   cd /root/wb-finance
   git pull
   ```
   (Если папка называется иначе — подставь свой путь к репозиторию.)

3. Пересобрать и запустить контейнеры:
   ```text
   docker compose down
   docker compose up -d --build
   ```

4. Проверить в браузере: http://194.87.96.144:8000/health и http://194.87.96.144:8000/docs

---

## Вариант 2: Первый раз развернуть с Git на сервере

Если на сервере ещё нет папки с проектом, а репозиторий уже есть в интернете:

1. Зайти на сервер: `ssh root@194.87.96.144`
2. Установить git (если нет): `apt update && apt install -y git`
3. Клонировать репозиторий (подставь свой URL):
   ```text
   cd /root
   git clone https://github.com/ТВОЙ_ЛОГИН/wb-finance.git
   cd wb-finance
   ```
4. Запустить: `docker compose up -d --build`

Дальше при обновлениях — только `git pull` в этой папке и снова `docker compose up -d --build`.

---

## Кратко (обычное обновление)

1. У себя: `git add .` → `git commit -m "..."` → `git push`
2. На сервере: `cd /root/wb-finance` → `git pull` → `docker compose down` → `docker compose up -d --build`
3. Проверить `/health` и `/docs`

---

## Если `git pull` пишет: `fatal: not a git repository`

Значит текущая папка на сервере **не клон из Git** (часто так бывает после **zip / scp / rsync**): в ней **нет** каталога `.git`, поэтому обновлять код через `git pull` **нельзя**, пока не сделаешь нормальный clone.

**Безопасный вариант (рекомендуется):**

1. Сохрани секреты (обязательно `backend/.env`):
   ```bash
   cp /root/wb-finance/backend/.env /root/wb-finance.env.backup
   ```
2. Клонируй репозиторий **в новую** папку (подставь свой URL репо):
   ```bash
   cd /root
   git clone https://github.com/chivkunovd-bitdenis/Wb_Finance.git wb-finance-git
   cp /root/wb-finance.env.backup wb-finance-git/backend/.env
   ```
3. Если у тебя уже собранный фронт лежал в старой папке — скопируй `frontend/dist` в новый клон (или собери на сервере по `DEPLOY.md`).
4. Останови старые контейнеры из старой папки (если они ещё запущены), подними из новой:
   ```bash
   cd /root/wb-finance-git
   # Сначала посмотри, какие сервисы реально есть в этом compose (имена могут отличаться от инструкции):
   docker compose config --services
   # Подними ВСЁ, что описано в файле — без перечисления имён (так не ошибёшься в celery_beat / и т.п.):
   docker compose up -d --build
   ```
   Если в выводе `docker compose config --services` **нет** `celery_beat`, значит на сервере старый `docker-compose.yml`. Сделай `git pull origin main` (после того как в GitHub залит актуальный compose) или скопируй актуальный `docker-compose.yml` с машины разработки.

   Если после `up` контейнеры не стартуют — смотри причину:
   ```bash
   docker compose ps -a
   docker compose logs --tail=80 api
   ```
5. Когда убедишься, что всё ок, **старую** `/root/wb-finance` можно переименовать в архив, а `wb-finance-git` переименовать в `wb-finance` — или оставить имя `wb-finance-git` и дальше всегда `cd` туда.

**Правило на будущее:** на проде держим **один** каталог = **git clone**, обновления только `git pull` + `docker compose up -d --build`.
