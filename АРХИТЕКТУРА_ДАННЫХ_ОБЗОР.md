# Обзор архитектуры хранения данных (GAS → PG)

Проверка текущей схемы в коде Google Apps Script и соответствие нашей PostgreSQL-схеме: нет ли лишнего, всего ли хватает для скорости.

---

## Что есть в GAS (листы в персональной таблице DB_<user>)

| Лист | Назначение | Источник данных | Ключ обновления |
|------|------------|-----------------|------------------|
| **DB_Raw_Data** | Сырые продажи | WB reportDetailByPeriod | date_from |
| **DB_Ads_Raw** | Сырая реклама (по nm_id после разбивки по кампаниям) | WB adv/v1/upd + детали кампаний | updTime |
| **DB_Articles** | Артикулы + себестоимость | Из продаж (каталог) + ручное редактирование | user + nm_id |
| **DB_PnL** | **Витрина**: P&L по дням (выручка, комиссия, логистика, штрафы, реклама, хранение, себестоимость, налог, маржа) | Пересчёт из RAW + ADS + Articles | user + date |
| **DB_Funnel_Daily** | Воронка по дням и артикулам | WB seller-analytics воронка | user + date + nm_id |

Плюс **Users** — один общий лист в главной таблице (логин, wb_api_key), не в персональной DB.

---

## Кто что читает в GAS

- **Дашборд (графики по дням)** — `apiGetPnlHistory` / дашборд-пэйлоад: читают **только DB_PnL**. Быстро, без пересчёта.
- **P&L пересчёт** — `updatePnlForPeriod`: читает RAW + ADS + Articles за период, пишет в DB_PnL. Вызывается после синка и после смены себестоимости.
- **Отчёт по артикулам (SKU)** — `apiGetSkuHistory`: читает **RAW + ADS + Funnel + Articles**, агрегирует в памяти (date + nm_id → продажи, реклама, маржа). Нет готовой витрины.
- **Time-series (графики по артикулам во времени)** — `apiGetTimeSeriesPayload`: то же — RAW + ADS + Funnel + Articles, агрегация в памяти.

Итого: витрина по **дням** (PnL) есть и используется. Витрины по **артикулам × дни** нет — каждый раз пересчёт из сырцов в памяти.

---

## Оценка: оптимальность в GAS

- **Лишних таблиц нет.** RAW, ADS, Articles, PnL, Funnel — у каждой своя роль; дублирования слоёв нет.
- **Витрина для скорости есть:** DB_PnL. Без неё каждый открытие дашборда требовало бы полного пересчёта из RAW/ADS — это правильно вынесено в отдельный слой.
- **Узкое место:** отчёты по артикулам (SKU, time-series) каждый раз считаются из сырцов в памяти. При росте объёма (много дней, много nm_id) это будет тормозить. В рамках Sheets отдельную витрину «по артикулам по дням» обычно не делают из-за ограничений и сложности обновления.

## Соответствие нашей PostgreSQL-схемы

| GAS | PostgreSQL | Комментарий |
|-----|------------|-------------|
| Users | users | Ок |
| DB_Raw_Data | raw_sales | Ок |
| DB_Ads_Raw | raw_ads | Ок |
| DB_Articles | articles | Ок |
| DB_PnL | pnl_daily | Ок, витрина по дням |
| DB_Funnel_Daily | funnel_daily | Ок |

Схема в PG **полностью соответствует** текущей логике GAS: те же слои, та же идея (сырьё → витрина PnL по дням).

---

## Рекомендации и что сделано

1. **Витрина по дням (pnl_daily)** — есть; дашборд по дням быстрый.
2. **Витрина «артикул × день» (sku_daily) добавлена.** Таблица `sku_daily`: user_id, date, nm_id, revenue, commission, logistics, penalties, storage, ads_spend, cogs, tax, margin, open_count, cart_count, order_count, order_sum. Заполняется задачей **recalculate_sku_daily** (агрегация из raw_sales, raw_ads, funnel_daily, articles). Эндпоинты time-series по SKU могут читать из неё одним SELECT.
3. **recalculate_pnl** в Celery — аналог `updatePnlForPeriod` (следующий шаг). После него цепочка: sync → recalculate_pnl → recalculate_sku_daily (вызывать после синка за тот же период).

---

## Краткий вердикт

Архитектура в GAS построена разумно: сырые данные, одна витрина PnL по дням, без избыточных таблиц. Наша PG-схема это повторяет. Дополнительные витрины для скорости имеет смысл вводить только при реальной необходимости (медленные отчёты по артикулам).
