# WB Finance: переход с Google Apps Script на свой сервер

Краткое описание того, что сделано, зачем, на каком этапе проект и что дальше.

---

## Что за проект

**WB Finance** — SaaS для финансовой аналитики продавцов на маркетплейсе Wildberries. Пользователь подключает API-ключ WB, система забирает данные о продажах, рекламе и воронке, считает P&L, маржу, ROI по товарам и показывает всё в дашборде.

---

## Откуда переходили и зачем

**Было:** рабочий прототип на **Google Apps Script (GAS)** + HTML/JS. Бэкенд и логика в GAS, данные в **Google Sheets**, фронт — один HTML-файл с вкладками (Дашборд, Артикулы, Себестоимость). Прототип использовали реальные пользователи.

**Зачем переходим на свой сервер:**
- Полный контроль над данными и логикой.
- Нормальный стек под рост: PostgreSQL, фоновые задачи, REST API, потом отдельный фронт на React.
- Возможность использовать нейросети и внешние API (на российском хостинге были ограничения).
- Масштабируемость и деплой на своём VPS.

---

## Что уже сделано (по шагам)

### 1. Инфраструктура (Docker Compose)

- Подняты сервисы одной командой `docker compose up`:
  - **PostgreSQL** — база данных.
  - **Redis** — брокер для фоновых задач.
  - **FastAPI** — бэкенд (API).
  - **Celery worker** — фоновые задачи (синк с WB, пересчёт P&L и т.д.).
- Проверка: после запуска в браузере открывается `http://IP:8000/health` → ответ `{"status":"ok"}`.

### 2. Схема базы данных и миграции

- Описаны **модели** под твою схему:
  - **users** — пользователи (email, пароль, API-ключ WB).
  - **articles** — артикулы и себестоимость.
  - **raw_sales** — сырые продажи с WB.
  - **raw_ads** — сырые данные по рекламе.
  - **pnl_daily** — P&L по дням.
  - **funnel_daily** — воронка по дням и артикулам.
- Настроен **Alembic** — миграции для PostgreSQL.
- При старте API миграции применяются автоматически (таблицы создаются в БД). Подробнее — в разделе «Как устроены миграции» ниже.

### 3. План разработки

- Составлен **план** в файле `ПЛАН_РАЗРАБОТКИ.md`: какие модули делать по очереди, как проверять каждый шаг, когда подключать автотесты.
- Решено: **миграции не трогаем** до момента, когда понадобятся таблицы (авторизация); логика — как в текущем коде (воронка 7 дней и т.д.).

### 4. Перенос на новый сервер

- Ты арендовал сервер в РФ — с него нельзя ходить в нейросети.
- Добавлен другой сервер (например 194.87.96.144), на него перенесён проект:
  - Архив папки проекта скопирован через **SCP** (логин **root**).
  - На сервере: распаковка архива, запуск `docker compose up -d --build`.
- Сейчас проект **работает на новом сервере** (здоровье проверяется через `/health`).

## На каком моменте сейчас

- **Работает:** Docker, PostgreSQL, Redis, FastAPI, Celery; миграции при старте API; проект развёрнут на новом VPS.
- **Не сделано:** авторизация (регистрация/логин/JWT), синхронизация с WB (Celery-задачи), расчёт P&L в новом коде, REST-эндпоинты для фронта, фронт на React.
- **Старый прототип:** GAS + HTML/JS и Google Sheets по-прежнему можно использовать; новый стек развиваем параллельно, пока не перенесём весь функционал.

## Как устроены миграции (что значит «мигрируют при старте API»)

### Что происходит при старте API

В коде бэкенда (`backend/app/main.py`) при запуске приложения вызывается одна функция:

- Событие **startup** (FastAPI поднялся).
- Функция **run_migrations()**: она берёт конфиг Alembic, подставляет адрес БД из переменной `DATABASE_URL` и выполняет команду **«применить все миграции до последней версии»** (`alembic upgrade head`).

То есть **«миграции при старте»** = каждый раз, когда контейнер с API запускается, перед приёмом запросов выполняется: «довести схему БД до актуального состояния по всем миграциям».

### Откуда берётся понимание структуры данных

Структура задаётся не «из воздуха», а из двух мест в нашем проекте.

**1. Модели в коде (Python)**  
В папке `backend/app/models/` лежат файлы с классами: `User`, `Article`, `RawSale`, `RawAd`, `PnlDaily`, `FunnelDaily`. В них описано:
- какая таблица в БД (имя),
- какие колонки (id, email, date, nm_id и т.д.),
- типы (UUID, строка, число, дата),
- связи (например, `user_id` → `users.id`).

Эти модели нужны **приложению**: по ним пишут и читают данные из БД в коде. Alembic при необходимости может смотреть на них (через `Base.metadata`), чтобы генерировать новые миграции, но **создание таблиц при первом запуске** у нас идёт из пункта 2.

**2. Файлы миграций (история изменений схемы)**  
В папке `backend/alembic/versions/` лежат файлы миграций. Первая миграция — `001_initial_tables.py`. В ней **вручную прописано**, что сделать с БД:
- создать таблицу `users` с колонками id, email, password_hash, wb_api_key, is_active, created_at;
- создать таблицу `articles` с колонками и связью с `users`;
- то же для `raw_sales`, `raw_ads`, `pnl_daily`, `funnel_daily` — индексы и ограничения.

То есть **понимание структуры при применении миграций** берётся именно из этих файлов: в них записано «создать такие таблицы с такими полями». Мы один раз перенесли твою схему (из описания и моделей) в этот файл — дальше Alembic только выполняет инструкции из файлов.

**Как это работает по шагам:**

1. Запускается контейнер API → вызывается `run_migrations()`.
2. Запускается Alembic с командой `upgrade head`.
3. Alembic смотрит в папку `alembic/versions/`, видит миграцию `001` (у неё нет предыдущей версии — она первая).
4. В БД есть служебная таблица `alembic_version` — в ней записано, какие миграции уже применены.
5. Если `001` ещё не применена → выполняется функция `upgrade()` из файла `001_initial_tables.py` → в PostgreSQL выполняются команды создания таблиц (CREATE TABLE users, CREATE TABLE articles, …).
6. В `alembic_version` записывается, что миграция `001` применена.
7. При следующих запусках API Alembic снова проверяет `alembic_version`, видит, что `001` уже есть, и ничего не делает — таблицы уже созданы.

Итого: **структура данных** задаётся (а) моделями в коде для работы приложения и (б) файлами миграций для приведения реальной БД к этой структуре. При старте API просто выполняется «применить все неприменённые миграции» — оттуда и появляются таблицы в PostgreSQL.
