# План разработки WB Finance (перенос на новый стек)

Чёткий порядок: что делаем, по одному модулю, как проверяем. Автотесты — когда и зачем.

---

## Принципы

- **Один модуль за раз** — сделали, проверили, переходим дальше.
- **Проверка после каждого шага** — иначе не поймёшь, где сломалось.
- **Автотесты** — добавляем после того, как основной скелет и ключевая логика уже работают (см. раздел про тесты ниже).

---

## Этап 0. Инфраструктура (уже сделано)

| Шаг | Что сделано | Как проверили |
|-----|------------|----------------|
| 0.1 | Docker Compose: PostgreSQL, Redis, API, Celery | `docker compose up --build` → в логах «Uvicorn running», «celery ready» |
| 0.2 | GET /health | В браузере `http://localhost:8000/health` → `{"status":"ok"}` |

**Миграции** — пока не трогаем. Подключим, когда дойдём до модуля, которому нужны таблицы (авторизация).

## Этап 1. Миграции и БД

| Шаг | Что делаем | Как проверяем |
|-----|------------|----------------|
| 1.1 | Запуск миграций при старте API (уже есть) или одной командой | После `docker compose up` в логах API нет ошибок; в БД появляются таблицы (через любой клиент к PostgreSQL или отдельной командой) |
| 1.2 | Краткая инструкция: как посмотреть, что таблицы созданы | Опционально: команда типа `docker compose exec postgres psql -U wb_finance -d wb_finance -c "\dt"` — список таблиц |

**Итог этапа:** в PostgreSQL есть все таблицы по схеме (users, articles, raw_sales, raw_ads, pnl_daily, funnel_daily). Дальше модули только пользуются ими.

## Этап 2. Авторизация

| Шаг | Что делаем | Как проверяем |
|-----|------------|----------------|
| 2.1 | Регистрация: POST /auth/register (email, password, wb_api_key) | Сохраняем пользователя в `users`, пароль — хэш. Проверка: запрос из браузера/Postman → 201, в БД одна запись в `users` |
| 2.2 | Вход: POST /auth/login (email, password) → JWT | Проверка: запрос с правильным паролем → 200 и токен в ответе; с неправильным → 401 |
| 2.3 | Защищённый эндпоинт: GET /auth/me (заголовок Authorization: Bearer <token>) | Проверка: с токеном → 200 и данные пользователя; без токена/с битым токеном → 401 |

**Итог этапа:** можно зарегистрироваться, войти и получать JWT; есть хотя бы один защищённый роут. Фронт пока не трогаем — проверяем через запросы (браузер, Postman или curl).

---

## Этап 3. Синхронизация с WB (Celery)

Каждую задачу делаем отдельно и проверяем, что она реально что-то делает (логи, запись в БД).

| Шаг | Что делаем | Как проверяем |
|-----|------------|----------------|
| 3.1 | Задача `sync_sales`: запрос к WB API (reportDetailByPeriod), запись в `raw_sales` | Ручной запуск задачи для тестового user_id + API-ключ; смотрим логи воркера и появление строк в `raw_sales` |
| 3.2 | Задача `sync_ads`: запрос к WB (adv/v1/upd + детали кампаний), запись в `raw_ads` | Аналогично: запуск задачи → логи + строки в `raw_ads` |
| 3.3 | Задача `sync_funnel`: воронка за 7 дней (как в текущем коде), запись в `funnel_daily` | Запуск задачи → логи + строки в `funnel_daily` |
| 3.4 | Триггер при логине (или по кнопке): запуск sync_sales + sync_ads параллельно, по готовности — recalculate_pnl (см. этап 4) | После «входа» (или по API) в логах Celery видны задачи; в БД обновляются raw_sales, raw_ads, потом pnl_daily |

**Итог этапа:** данные с WB попадают в БД; воронка — отдельно (по кнопке/отдельный вызов), как в прототипе.

---

## Этап 4. Расчёт P&L

| Шаг | Что делаем | Как проверяем |
|-----|------------|----------------|
| 4.1 | Задача или сервис `recalculate_pnl`: читает raw_sales, raw_ads, articles (себестоимость) за период, считает по формулам из кода (налог 6%, комиссия, маржа и т.д.), пишет в `pnl_daily` | Запуск после sync_sales/sync_ads; сверка с текущим прототипом (те же цифры за те же даты) |
| 4.2 | Связка: после sync_sales и sync_ads автоматически дергать recalculate_pnl за обновлённый период | Проверка: полный цикл «синк → пересчёт» без ручного вызова recalculate_pnl |

**Итог этапа:** P&L считается так же, как в GAS; данные в `pnl_daily` совпадают с прототипом на одних и тех же данных.

## Этап 5. REST API для фронта

Эндпоинты только отдают уже посчитанные данные из БД (без дублирования сложной логики в API).

| Шаг | Что делаем | Как проверяем |
|-----|------------|----------------|
| 5.1 | GET /api/dashboard (или /api/pnl) — P&L по дням за период, по user_id из JWT | Запрос с токеном + date_from, date_to → 200, массив записей из `pnl_daily` |
| 5.2 | GET /api/sku (или /api/articles/stats) — агрегат по артикулам (выручка, маржа, реклама и т.д.) за период | Аналогично: токен + период → данные для вкладки «Артикулы» |
| 5.3 | GET /api/articles — список артикулов и себестоимость; PATCH/PUT — обновить себестоимость | Чтение и изменение записей в `articles` |
| 5.4 | GET /api/funnel — воронка за период (из `funnel_daily`) | Токен + период → данные для вкладки «Воронка» |
| 5.5 | POST /api/sync (или кнопка «Обновить WB») — постановка в очередь: sync_sales + sync_ads + recalculate_pnl за выбранный период | Запрос → 202 Accepted; в логах Celery видны задачи |

**Итог этапа:** фронт (даже будущий React) может получать все данные и запускать синхронизацию через API; проверка — запросы с JWT (Postman/curl/браузер).

## Этап 6. Фронтенд (React)

| Шаг | Что делаем | Как проверяем |
|-----|------------|----------------|
| 6.1 | Каркас: Vite + React, роуты, экран логина (форма → POST /auth/login, сохранение JWT) | Вход по email/пароль → редирект в приложение, токен в localStorage/cookie |
| 6.2 | Вкладка «Дашборд»: период, запрос к /api/dashboard, таблица и графики (как в текущем HTML) | Визуально те же метрики, что в прототипе |
| 6.3 | Вкладки «Артикулы», «Себестоимость», «Воронка» — запросы к соответствующим API, отображение таблиц | Данные совпадают с тем, что отдают эндпоинты |
| 6.4 | Кнопка «Обновить WB» → POST /api/sync, индикатор загрузки, обновление данных после завершения (опрос статуса или webhook — как решим) | Нажал → задачи в Celery, данные на фронте обновились |

**Итог этапа:** полный цикл в браузере: вход, дашборд, артикулы, себестоимость, воронка, обновление с WB.

## Автотесты: когда и что

- **Сразу всё покрывать тестами не обязательно** — особенно пока нет опыта в программировании. Сначала доводим модули до рабочего состояния и проверяем вручную.
- **Когда имеет смысл добавлять:**
  - **После этапа 4 (P&L):** unit-тесты на формулы (на входе — сырые строки продаж/рекламы, на выходе — ожидаемые суммы в pnl_daily). Так мы зафиксируем логику и не сломаем её при изменениях.
  - **После этапа 5 (REST):** несколько интеграционных тестов (pytest): вызов эндпоинтов с тестовым пользователем и тестовой БД — проверка, что API возвращает 200 и ожидаемую структуру.
  - **Фронт (Jest):** позже, когда React-приложение стабильно — можно добавить тесты на критические сценарии (логин, загрузка дашборда).
- **Итог:** сначала делаем модули и проверяем вручную; автотесты подключаем после того, как основная логика и API уже работают — в первую очередь на расчёт P&L и на ключевые эндпоинты.

## Порядок действий (кратко)

1. **Миграции** — включить и проверить, что таблицы есть (можно отложить до этапа 2, но тогда к авторизации уже должна быть готова БД).
2. **Авторизация** — регистрация, логин, JWT, /auth/me.
3. **Celery:** sync_sales → sync_ads → sync_funnel (по одной задаче, каждую проверяем).
4. **P&L:** recalculate_pnl и связка с синком.
5. **REST API** — эндпоинты для дашборда, артикулов, себестоимости, воронки, синка.
6. **Фронт React** — логин, дашборд, артикулы, себестоимость, воронка, кнопка обновления.
7. **Автотесты** — после этапов 4–5: формулы P&L и основные запросы к API.

Миграции можно оставить «включёнными» при старте API (как сейчас) и просто один раз убедиться, что после `docker compose up` таблицы созданы. Дальше двигаемся по плану: следующий атомарный шаг — **авторизация**.
