# New Feature

## Когда использовать
Для разработки новой функции, новой страницы, нового сценария или нового поведения.

## Цель
Реализовывать новую функциональность безопасно, по шагам, без лишнего хаоса.

## Обязательный процесс

1. Сначала понять задачу и контекст: требования, ожидаемое поведение, ограничения, критерии “готово”.
2. Если фича затрагивает пользовательский сценарий/API/фоновые задачи/БД/state/очереди/UI — до кода составить `Scenario contract` и `Flow trace` через global skill `scenario-contract`.
3. Найти затронутые зоны кода:
   - какие модули/компоненты нужно тронуть
   - какие места трогать не нужно
4. Составить короткий пошаговый план (несколько шагов) до любых правок в коде.
5. Реализовать изменения минимально и поэтапно:
   - сначала “скелет/контуры”, затем “логика”, затем “интеграция”
   - избегать попутных изменений за пределами задачи
6. Не делать большой попутный рефакторинг:
   - допускается только точечное исправление, необходимое для новой функциональности
7. Добавить или обновить необходимые тесты:
   - покрыть новый сценарий и ключевые ветки
   - при наличии существующих тестов — расширить их, а не заменять целиком
   - для сценарной фичи: user action -> API/фон -> DB/state -> visible result
8. Провести самопроверку перед отдачей результата:
   - проверить ожидаемое поведение вручную (если применимо)
   - убедиться, что тесты проходят
   - для сценарной фичи: приложить `Scenario proof`; без него не писать “готово”
9. Зафиксировать задачу в `TASKLOG.md` и привязать к коммиту:
   - добавить строку `TASK-<N>` (N+1), дата, название
   - “что менялось” и “что НЕ менялось”
   - после верификации сделать commit и вписать hash в `TASKLOG.md`
10. Кратко описать итог в формате отчёта:
   - что сделано
   - что проверено
   - какие риски остались

## Ограничения
- Не менять архитектуру без необходимости.
- Не трогать несвязанные части проекта.
- Не считать задачу завершённой без проверки (тесты и/или самопроверка по чеклисту).

## На выходе должно быть
- Короткий пошаговый план до кодинга.
- Scenario contract/proof (если фича сценарная).
- Список затронутых зон кода (и список “не трогаем”).
- Итоговый отчёт: “что сделано / что проверено / какие риски”.
