# Write Tests

## Когда использовать
Когда задача требует автоматической проверки поведения, покрытия новой логики или защиты от регрессии.

## Цель
Добавить полезные автоматические тесты без избыточности и хрупкости.

## Обязательный процесс

1. Определить уровень тестов:
   - `unit` для логики без внешних зависимостей
   - `integration` для взаимодействий между слоями (например, API + БД, сервисы + репозитории)
   - `e2e` только если без полного сценария нельзя убедиться в корректности

2. Покрыть ключевое поведение и edge cases:
   - позитивные сценарии (ожидаемый результат)
   - негативные сценарии (ошибки/валидация/границы)
   - хотя бы один тест на “почему это важно” (регрессионный смысл)
   - для сценарных изменений: user action -> API/фон -> DB/state -> visible result

3. Избегать хрупких тестов:
   - не проверять несущественные детали реализации (внутренние переменные/точные строки сообщений, если это не контракт)
   - не использовать тайминги, “ожидание по секундам”, случайные данные без фиксации
   - не зависеть от порядка выполнения тестов

4. Явно зафиксировать покрытие:
   - что покрыто (и на каких уровнях: unit/integration/e2e)
   - что сознательно не покрыто и почему (ограничения/стоимость/риск незначительный)

5. Для queue/retry/background-job изменений проверить отсутствие дублей/шторма и error/retry path, либо явно указать пробел.

6. Обязательно прогнать тесты перед завершением:
   - выполнить локально набор тестов, который относится к изменениям (минимум relevant subset)
   - убедиться, что тесты детерминированные

## Ограничения
- Не заменять тестами отсутствие нормальной логики.
- Не плодить тяжёлые `e2e` без причины.

## На выходе
- Описание: какие тесты добавлены/обновлены и почему это именно тот уровень.
- Список покрытых сценариев (positive/negative/edge cases).
- Scenario contract coverage / proof gaps (если применимо).
- Результат запуска: какие тесты прошли.
- Остаточный risk: что может сломаться и как это будет обнаружено в будущем.
