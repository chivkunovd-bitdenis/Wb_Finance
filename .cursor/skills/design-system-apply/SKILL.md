# Design System Apply

## Когда использовать
При реализации нового интерфейса или обновлении существующего визуального слоя.

## Цель
Делать интерфейсы визуально целостными и единообразными.

## Обязательный процесс

1. Использовать единый visual baseline из файла `Dashboard.html`.
   - Если `Dashboard.html` в репозитории отсутствует, использовать ближайший референс дизайна проекта (например `Index.html` или главный экран-дешборд в проекте).

2. Сохранять единый ритм отступов, размеров и плотности:
   - spacing: брать значения из baseline и не “изобретать” новые
   - размеры: сохранять иерархию (title/subtitle/body)
   - плотность: не увеличивать визуальный шум (слишком плотные/широкие сетки запрещены)

3. Сохранять единый стиль карточек, таблиц, форм и кнопок:
   - карточки: одинаковые `radius-card`, border и внутренние отступы
   - таблицы: одинаковые заголовки/строки/плотность, единый стиль выделения
   - формы: единые поля/лейблы, одинаковые `radius-element`, border
   - кнопки: единый вариант (primary/secondary/danger) и hover/focus стили

4. Не смешивать случайные визуальные паттерны:
   - не комбинировать разные радиусы/бордеры/шрифтовые размеры в одной сущности
   - не использовать “локальные” стили, если уже есть общий паттерн/класс/переменная

5. Проверять, что экран выглядит как часть одной системы:
   - сравнить визуально новый/измененный экран с baseline по ключевым блокам
   - пройтись по состояниям: normal/hover/focus/error/loading (где применимо)

## Дизайн-токены проекта

- accent: `#5b4fd4`
- accent-mid: `#7c6ff7`
- accent-light: `#f0effe`
- positive: `#16a34a`
- negative: `#dc2626`
- border: `0.5px solid rgba(0,0,0,0.08)`
- radius-card: `12px`
- radius-element: `8px`

## Ограничения
- Не вводить новый стиль без необходимости.
- Не делать визуальный шум.

## На выходе
- Кратко: чем новый/измененный UI соответствует baseline.
- Что переиспользовано (классы/компоненты/токены).
- Какие точки проверки выполнены (ритм, карточки/таблицы/формы/кнопки, основные состояния).
