# Как запустить автотесты

Используется **pytest** — стандартный инструмент для тестов в Python.

---

## Где что лежит

- **Тесты:** папка `backend/tests/`
  - `test_security.py` — хэш пароля, JWT (без сети и БД)
  - `test_wb_client.py` — запросы к WB с подменой HTTP (mock)
  - `test_auth_api.py` — регистрация, логин, /me (БД подменена на mock)
  - `test_sync_api.py` — POST /sync/sales, /ads, /funnel (Celery подменён на mock)
  - `test_tasks.py` — задачи recalculate_pnl (pnl_daily), recalculate_sku_daily (sku_daily); БД подменена на mock
  - `test_dashboard_api.py` — GET /dashboard/pnl, /articles, /funnel, /sku (с JWT и без)
  - `test_wb_client_real.py` — **интеграционные**: реальные запросы к WB (только при заданном `WB_API_KEY`)
- **Конфиг pytest:** `backend/pytest.ini` (путь к тестам и `pythonpath`)

---

## Запуск

Из папки **backend** (чтобы подхватить `pytest.ini` и путь к коду):

```text
cd backend
pytest
```

Или из корня проекта:

```text
cd /Users/deniscivkunov/wb-finance/backend
pytest
```

Прогнать один файл:

```text
pytest tests/test_security.py
```

Прогнать один тест:

```text
pytest tests/test_security.py::test_hash_password_returns_string
```

С кратким отчётом по покрытию:

```text
pytest --cov=app --cov-report=term-missing
```

---

## Что проверяют тесты

- **security:** один и тот же пароль даёт один и тот же хэш; проверка пароля ок/не ок; JWT создаётся и расшифровывается; битый токен → None.
- **wb_client:** при «фейковом» ответе WB разбор полей и пагинация; для рекламы — деление расхода по артикулам.
- **auth_api:** регистрация возвращает 200 и данные пользователя; логин с верным паролем — токен; с неверным — 401; /me без токена — 401.
- **sync_api:** с валидным JWT вызов POST /sync/sales и /sync/ads возвращает 200 и task_id; Celery не вызывается по-настоящему (подменён mock).

Для большинства тестов реальная БД, Redis и WB API не нужны.

Но часть **интеграционных** тестов использует `real_db_session` и будет запускаться,
если доступна PostgreSQL по `DATABASE_URL` (или по дефолту приложения — localhost).

---

## Запуск тестов в Docker

Если не хочешь ставить Python и зависимости у себя на компе, можно прогнать тесты в контейнере (сначала собери образ: `docker compose build api` из корня проекта):

```text
cd /путь/к/wb-finance
docker compose run --rm api pytest tests/ -v
```

Команда запускает контейнер с тем же кодом, что и API, и выполняет в нём pytest.

### Важно: чтобы интеграционные тесты не падали

Интеграционные тесты требуют **Postgres** (и иногда Celery/Redis). Самый простой “без вопросов” способ:

```text
cd /путь/к/wb-finance
./scripts/run_tests_docker.sh
```

## Тесты с реальным WB API

В `test_wb_client_real.py` проверяется, что запросы к Wildberries действительно уходят с нужным периодом и ответ приходит в ожидаемом виде (структура полей, даты в диапазоне). Так можно сразу ловить перепутанные параметры периода или изменения формата ответа WB.

**Без ключа WB эти тесты пропускаются** (не падают). Ключ можно задать двумя способами:

### Вариант 1: файл backend/.env (удобно, ключ не нужно вводить каждый раз)

1. Скопируй пример и подставь свой ключ:
   ```text
   cd backend
   cp .env.example .env
   ```
2. Открой `backend/.env` и впиши ключ WB в строку `WB_API_KEY=...`
3. Файл `.env` в git не попадает (он в .gitignore).

Запуск тестов с реальным WB:

```text
# Локально из папки backend
pytest tests/test_wb_client_real.py -v

# В Docker (файл backend/.env монтируется в контейнер сам)
docker compose run --rm api pytest tests/test_wb_client_real.py -v
```

### Вариант 2: переменная окружения

```text
WB_API_KEY=твой_ключ pytest tests/test_wb_client_real.py -v
docker compose run --rm -e WB_API_KEY=твой_ключ api pytest tests/test_wb_client_real.py -v
```

Проверяется: `fetch_sales`, `fetch_ads`, `fetch_funnel` — тип ответа (список), наличие нужных полей у каждой записи, что даты попадают в запрошенный период (date_from..date_to).

Для воронки нужны артикулы (nm_id): тест берёт их из ответа продаж за тот же период. Если продаж нет, можно задать в `.env`: `WB_FUNNEL_NM_IDS=12345,67890`.
